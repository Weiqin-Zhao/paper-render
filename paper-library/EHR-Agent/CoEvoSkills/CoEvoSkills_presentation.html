1.1 Tool → Skill 的范式转变

1.1 Tool → Skill Paradigm Shift

LLM agent 的最近进展大量依赖 tool/API 调用（ToolLLM、Gorilla、Toolformer），但真正的专业任务（复杂软件修复、多步科研分析、企业数据管道）不是一次 tool 调用能解决的：决策 long-horizon、指令与脚本紧耦合、环境反馈稀疏。

Recent LLM-agent progress leans heavily on tool/API invocation (ToolLLM, Gorilla, Toolformer). But real professional tasks — complex software repair, multi-step scientific analysis, enterprise pipelines — require far more than one-shot tool calls: long-horizon decisions, tightly coupled instructions and scripts, and sparse environmental feedback.

Anthropic 提出 agent skill：一个 skill 不是单一函数，而是一个多文件结构化包（workflow 说明 + 可执行脚本 + 领域参考材料）——Figure 1 展示了 tool 和 skill 的结构差异。

Anthropic introduced agent skills: a skill is not a single function but a multi-file structured bundle (workflow instructions + executable scripts + domain references) — Figure 1 illustrates the structural difference.

SkillsBench 评估发现：人工 skill 收益极不均匀——某些领域（Natural Science）甚至 使 agent 性能下降。作者的假设：

The SkillsBench evaluation shows that gains from human-curated skills are highly uneven — in some domains (Natural Science) they actively degrade agent performance. The hypothesis:

人类专家设计的 workflow/抽象符合人类直觉，但不匹配 LLM agent 的上下文处理、推理模式和执行约束。
Workflows and abstractions designed to be intuitive for human experts do not match how LLM agents process context, reason, and act under execution constraints.

2.2 POMDP 形式化

2.2 POMDP Formulation

任务环境建模为 POMDP $\mathcal{M} = \langle \mathcal{X}, \mathcal{A}, T, \mathcal{O}, \Omega, \mathcal{R} \rangle$：$\mathcal{X}$ 是文件系统+进程状态，$\mathcal{A}$ 是终端命令+文件编辑，$\mathcal{O}$ 是命令执行结果（部分可观测），$\mathcal{R}(x_T) \in [0,1]$ 是对最终输出的隐藏 GT 评分。Skill 作为策略条件：

Model the task environment as a POMDP $\mathcal{M} = \langle \mathcal{X}, \mathcal{A}, T, \mathcal{O}, \Omega, \mathcal{R} \rangle$: $\mathcal{X}$ is filesystem+processes, $\mathcal{A}$ is shell commands+file edits, $\mathcal{O}$ is partial observations (command outputs), and $\mathcal{R}(x_T) \in [0,1]$ is the hidden GT reward on the terminal state. The skill conditions the policy:

$$a_t \sim \pi_\theta(a_t \mid h_t,\, \mathcal{S})$$

优化目标是找到最大化期望终局回报的 skill：

The objective is to find the skill that maximizes expected terminal reward:

$$\mathcal{S}^* = \arg\max_{\mathcal{S}}\; J(\mathcal{S}),\quad J(\mathcal{S}) \triangleq \mathbb{E}_{\tau}[\mathcal{R}(x_T)]$$

直接优化 $J$ 不可行——oracle 只返回 1-bit，没有梯度也没有诊断。

Directly optimizing $J$ is intractable — the oracle returns only 1 bit, no gradient, no diagnostic.

2.3 Surrogate Reward 与交替优化

2.3 Surrogate Reward & Alternating Optimization

引入独立 verifier 合成的 assertion 套件 $\mathcal{V}=\{e_1,\dots,e_{|\mathcal{V}|}\}$，给出 surrogate reward：

Introduce a verifier-synthesized assertion suite $\mathcal{V}=\{e_1,\dots,e_{|\mathcal{V}|}\}$, yielding the surrogate reward:

$$\tilde{\mathcal{R}}(x, \mathcal{V}) \triangleq \frac{1}{|\mathcal{V}|}\sum_{k=1}^{|\mathcal{V}|}\mathbf{1}[e_k(x)] \in [0,1]$$

然后交替优化：

Then alternate:

$$\textbf{Skill refinement:}\quad \mathcal{S}^{(i+1)} \leftarrow \arg\max_{\mathcal{S}}\; \tilde{\mathcal{R}}(\Phi(\mathcal{S},\mathcal{E}),\, \mathcal{V}^{(j)})$$
$$\textbf{Test escalation:}\quad \mathcal{V}^{(j+1)} \sim \pi_\theta^V(\cdot \mid I, x^{(i)}, \mathcal{V}^{(j)}),\; \text{if}\; \mathbf{1}[\tilde{\mathcal{R}}=1 \wedge \mathcal{R}<1]$$

Skill refinement 在固定 $\mathcal{V}^{(j)}$ 下改进 skill；test escalation 只在 surrogate 通过但 GT 失败时触发，只拿到 1-bit oracle 信号，没有 test content，因此 verifier 必须"独立"加强自己的 assertion（例如提高精度阈值、增加 assertion 数）。

Skill refinement improves the skill under a fixed $\mathcal{V}^{(j)}$; test escalation triggers only when surrogate passes but GT fails, seeing only the 1-bit oracle signal with no test content, so the verifier must independently strengthen its assertions (e.g. tighten precision thresholds, add assertions).

2.4 Co-Evolution Algorithm（Alg. 1）

2.4 Co-Evolution Algorithm (Alg. 1)

关键超参：$N=5$ evolution iters、$M=15$ surrogate iters、$\beta=0.7$ context cap。

Key hyperparameters: $N=5$ evolution iters, $M=15$ surrogate iters, $\beta=0.7$ context cap.

while n < N and r < M: x = Φ(S, E) # rollout skill in env if context_usage > β: break # prevent OOM R̃ = surrogate_reward(x, V) if R̃ < 1: # Surrogate failed F = verifier_diagnose(I, x, V) # per-assertion + root cause C = C ⊕ F S = refine(S, C); i++; r++; continue # Surrogate passed → invoke Oracle x̂ = Φ(S, E') # fresh env re-execution R = oracle(x̂); n++ if R == 1: return S elif R > R_best: save S as S* C = C ⊕ 𝟙[R<1] # ONLY 1-bit, no content V = escalate(I, x, V) # verifier must independently escalate return S*

Skill Generator 细节

Skill Generator Details

持久 context $C^{(0)}=(I, \mathcal{S}_{\text{meta}})$，其中 $\mathcal{S}_{\text{meta}}$ 是 Anthropic 官方的 skill-creator meta-skill。每次迭代

Persistent context $C^{(0)}=(I, \mathcal{S}_{\text{meta}})$, where $\mathcal{S}_{\text{meta}}$ is Anthropic's official skill-creator meta-skill. Each round:

$$\mathcal{S}^{(i+1)} \sim \pi_\theta(\cdot \mid \mathcal{S}^{(i)}, C^{(i+1)}),\quad C^{(i+1)} = C^{(i)} \oplus \mathcal{F}^{(i,j)}$$

$\mathcal{F}^{(i,j)}$ 是 verifier 的结构化诊断：failed tests + root-cause + 修复建议；$\oplus$ 是 context 追加。

$\mathcal{F}^{(i,j)}$ is the verifier's structured diagnostic (failed tests + root-cause + revision suggestions); $\oplus$ is context append.

Surrogate Verifier 细节

Surrogate Verifier Details

  • 独立 LLM session（不看 Generator 的 reasoning / code / skill 内容）
  • 只观察 task instruction $I$ 和 output files $x^{(i)}$ → 避免 confirmation bias
  • 输出：deterministic assertion 套件 + per-assertion 诊断（结果 / root cause / 修复建议）
  • Independent LLM session (blind to generator's reasoning/code/skill)
  • Observes only task instruction $I$ and output files $x^{(i)}$ → avoids confirmation bias
  • Output: deterministic assertion suite + per-assertion diagnostic (result / root cause / suggested fix)

CoEvoSkills outperforms human-curated skills in 9/11 domains:

Takeaway 3: Human-curated skills can actively degrade performance in some domains; co-evolution bridges the gap by discovering procedures aligned with how LLMs actually reason.

Surrogate vs. GT 的固有 gap（极好的反例）

Surrogate vs. GT gap (excellent counterexample)

Round 3：surrogate 15/15 全过，GT 却 3/4。Surrogate 用 1% tolerance 匹配周期，GT 要求 5-decimal 精确匹配——surrogate 无法从 task 描述推断出这个阈值。

Round 3: surrogate 15/15, but GT 3/4. Surrogate used a 1% tolerance while GT required 5-decimal exact match — the surrogate cannot infer this threshold from the task description alone.

Round 5：Surrogate 升到 22 tests，自己的 BLS 算出 3.24158，agent 的 TLS 算出 3.24156。Surrogate 把 0.00002 天的差标成 fail——虽然 agent 的答案其实更准。

Round 5: the surrogate's own BLS produced 3.24158 while the agent's TLS produced 3.24156. The surrogate flagged the 0.00002-day discrepancy as failure — even though the agent's answer was in fact more accurate.

这说明 surrogate 的两个结构性局限：(1) 无法复刻 GT 的精度要求；(2) 无法区分 "自己估计的误差" 与 "agent 的误差"。因此 GT Oracle 不可被 surrogate 取代——必须作为权威 arbiter。
This reveals two structural limitations: (1) the surrogate cannot replicate GT's exact precision; (2) it cannot distinguish its own estimation error from the agent's. The GT Oracle is irreplaceable as the authoritative arbiter.

讨论与局限

Discussion & Limitations

为什么 Surrogate Verifier 这么关键？ Why is the Surrogate Verifier so critical?
消融显示 surrogate 贡献 30pp。本质上它的价值是：(1) 把 1-bit GT 信号扩展成 per-assertion 诊断 → targeted repair 成为可能；(2) 吸收 ~40% 的 oracle 预算 → 节省真实评测成本；(3) 信息隔离 → 避免 Generator 的 self-verification bias。
Ablations show the surrogate contributes 30pp. Its value: (1) expands the 1-bit GT signal into per-assertion diagnostics → targeted repair becomes possible; (2) absorbs ~40% of the oracle budget → saves real-eval cost; (3) information isolation → avoids the generator's self-verification bias.
信息隔离如何防止对 held-out test 过拟合？ How does information isolation prevent held-out overfitting?
设计上 oracle 只返回 $\mathbf{1}[\mathcal{R}<1]$，不返回失败内容；verifier 必须靠 task instruction 和 output 独立合成 test。Case study round 5 是一个正面验证——surrogate 甚至比 GT 更严格（把比 GT 更准的答案标成 fail），这不可能出现在"偷看 held-out test"的 setting 里。
By design the oracle returns only $\mathbf{1}[\mathcal{R}<1]$, no failure content; the verifier must synthesize tests from the task instruction and outputs alone. Round 5 of the case study is positive evidence — the surrogate flagged an answer even more accurate than GT as failure, impossible if it were peeking at held-out tests.
Budget 选择与 sensitivity Budget selection & sensitivity
$N=5$、$M=15$ 怎么选的？失败的 10 个任务集中在 budget 用光处，说明 budget 可能是性能瓶颈——论文没做 sensitivity 分析。如果把 $N$ 放到 10 会怎样？增益曲线形状如何？这是复现者首先会问的问题。
How were $N=5$, $M=15$ chosen? The 10 failed tasks cluster at exhausted budget, suggesting budget may be a bottleneck — but no sensitivity study is given. What happens at $N=10$? What does the gain curve look like? This is the first question a re-implementer would ask.
Surrogate overshoot 的代价 The cost of surrogate overshoot
Round 5 那种"surrogate 比 GT 还严格"的情形在多少任务上发生？论文没定量。理论上 surrogate 误报会浪费 evolution 轮次、把好 skill 改坏。没有定量衡量意味着 verifier 是否始终"helpful"存疑。
How often does "surrogate stricter than GT" happen across tasks? Not quantified. A too-strict surrogate burns evolution rounds and can regress a good skill — without this measurement we cannot claim the verifier is always helpful.
单一 benchmark 与 ablation 单次 run Single benchmark, single-run ablations
作者承认 SkillsBench 是"唯一 skill-purpose benchmark"，但 87 tasks × 7 模型 × 多 baseline × 3-5 seeds 的矩阵统计力是否足够？ablation 都是单次 run（成本原因），置信区间缺失。只用一个 benchmark 的结论的外推性值得保留。
The authors note SkillsBench is the "only skill-purpose benchmark," but is 87 tasks × 7 models × multiple baselines × 3-5 seeds enough statistical power? Ablations are single-run (cost reason) with no confidence intervals. Claims from a single benchmark should be held conservatively.
Cost transparency 弱 Weak cost transparency
3000s/task × 4 workers × 5 runs × 5 iters × 多模型 = 相当昂贵。论文未给 token cost 或 $ cost，对复现不友好；对读者判断"这个框架的 cost-benefit 是否值得"也不友好。
3000s/task × 4 workers × 5 runs × 5 iters × multiple models = substantial cost. No token count or $ figure is given, hurting reproducibility and cost–benefit reasoning.
"Agent 比人好" 结论的边界 Boundary of the "agents beat humans" claim
人工 skill 是 SkillsBench 提供的那个版本——是一组特定人写的，不一定代表"人类能做的最好"。一个经验丰富的该领域专家花同样时间写是否能赶上？作者没对比。另外 skill-creator 本身是人写的 meta-skill，CoEvoSkills 依赖它 bootstrap——严格说仍是 human-in-the-loop at meta-level。
The "human-curated" version is the one shipped with SkillsBench — a specific group's output, not necessarily "the best humans can do." Would a seasoned domain expert writing for comparable time match it? Not tested. Also, skill-creator is itself a human-authored meta-skill that bootstraps CoEvoSkills — strictly, this is still human-in-the-loop at the meta-level.
Verifier 的脆弱性（Open-ended 任务） Verifier fragility (open-ended tasks)
如果 task 描述很模糊（现实常见），verifier 合成的 assertion 可能完全偏离 GT。论文挑的是 deterministic-verifier 任务；在 open-ended 任务（UI 设计、创作、研究写作）上 surrogate 会怎样？是这个框架最值得后续探索的方向。
With vague task descriptions (common in the real world), the verifier's assertions may drift from GT. This paper only considers deterministic-verifier tasks. How does the surrogate behave on open-ended tasks (UI design, creative writing, research writing)? The most important direction for future work.
可组合性（Skill Composability） Skill composability
如果一个任务需要多个进化过的 skill 组合，它们会不会互相矛盾？论文只评测了"一个任务一个 skill"。skill 之间的冲突检测和合并是未解决的开放问题。
What if a task needs multiple evolved skills composed? Could they conflict? The paper only evaluates one-skill-per-task. Cross-skill conflict detection and merging is an open problem.
Negative transfer 分析缺失 Missing negative-transfer analysis
Opus 进化的 skill 给 Haiku 用比 no-skill 高 44pp，但真的没有"某个 skill 让某个模型变差"的案例吗？论文只报了 domain-level aggregate，per-task × per-model 粒度上是否仍有负迁移？值得一张热图。
Opus-evolved skills lift Haiku by +44pp — but is there any (skill, model) pair with negative transfer? The paper reports domain-level aggregates; a per-task × per-model heatmap would be valuable.
