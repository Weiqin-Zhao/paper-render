📚 Primer: Prompt / Skill / Harness

读这篇论文前，先对齐三个经常被混淆的概念。 Before reading the paper, align three frequently-confused concepts. 它们不是并列关系，而是一层套一层的嵌套。 They're not siblings — they form a nested hierarchy.

0.1 一张图理解

0.1 The One-Picture View

CPU / OS / 应用 类比：原始 LLM 像一颗没 RAM、没硬盘、没 I/O 的 CPU。上下文窗口是 RAM（快但小），外部数据库是 硬盘（大但慢），工具集成是 设备驱动，而 harness 就是操作系统。

CPU / OS / App analogy: a raw LLM is a CPU with no RAM, no disk, no I/O. Context window = RAM (fast, small), external DB = disk (large, slow), tool integrations = device drivers, and harness = the operating system.

围绕模型有三层同心的工程：Prompt engineering（起草给模型的指令）⊂ Context engineering（管理模型看到什么、何时看到）⊂ Harness engineering（前两者 + 工具编排、状态持久化、错误恢复、验证循环、安全执行、生命周期管理）。

Three concentric layers of engineering around the model: Prompt engineering (drafts instructions) ⊂ Context engineering (manages what the model sees and when) ⊂ Harness engineering (the above + tool orchestration, state persistence, error recovery, verification loops, safe execution, lifecycle management).

0.3 Prompt（提示词）

0.3 Prompt

  • 对话里一次性的自然语言指令
  • A one-shot natural-language instruction in conversation
  • 灵活、快速，但不跨会话保留，重复劳动多
  • Flexible and fast, but doesn't persist — repetitive work
  • 适用：临时任务、头脑风暴、原型探索、一次性调整
  • Use for: one-off tasks, brainstorming, prototyping, ad-hoc tweaks

0.4 Skill（技能）

0.4 Skill

  • Anthropic 2025-10 推出，解决"同一个 prompt 反反复复写"的痛点
  • Announced by Anthropic Oct 2025 to solve the "typing the same prompt over and over" pain
  • 一个可复用指令集：SKILL.md + 元数据 + 可选资源（脚本/模板/参考文档），丢到 .claude/skills/ 即可
  • A reusable instruction package: SKILL.md + metadata + optional resources (scripts/templates/docs), dropped into .claude/skills/
关键机制：渐进式披露（progressive disclosure） Key mechanism: progressive disclosure Claude 启动时只看到 skill 的名字和描述，真正需要时才加载完整内容。这是 skill 和"把 prompt 存文本文件里"最大的区别——它不占 context 预算。 At startup Claude only sees each skill's name and description; full contents load only when needed. This is what separates skills from "just put your prompt in a file" — they don't eat your context budget.
Skill vs Prompt 决策原则（Anthropic 官方）： Skill vs Prompt decision rule (from Anthropic): "如果你发现自己在多个对话里一遍遍敲同一段 prompt，那就该做一个 skill 了。" "If you find yourself typing the same prompt across conversations, it's time to make a skill."

0.5 Harness（脚手架 / 载具）

0.5 Harness

  • 2026 年初才被"正式命名"，但概念早就存在
  • Formally named only in early 2026, but the concept has been around
  • 包裹 LLM 的完整软件基础设施：编排循环、工具、记忆、上下文管理、状态持久化、错误处理、护栏
  • The complete software infra wrapping an LLM: orchestration loop, tools, memory, context management, state persistence, error handling, guardrails
  • Anthropic 官方："Claude Code SDK 就是驱动 Claude Code 的 agent harness。"
  • Anthropic: "The Claude Code SDK is the agent harness that powers Claude Code."
  • LangChain 经典公式："如果你不是那个模型，那你就是 harness。"
  • LangChain's axiom: "If you're not the model, you're the harness."
为什么需要 harness？ Why do we need a harness? 因为 LLM 本质无状态：每个新会话都从零开始。Harness 管理上下文压缩、工具分发、会话持久化、进度追踪、验证循环、错误恢复、安全执行。 Because LLMs are fundamentally stateless: every session starts from scratch. The harness manages compaction, tool dispatch, session persistence, progress tracking, verification loops, error recovery, and safe execution.
反直觉但重要的洞察：Harness 会被"削薄" Counter-intuitive but important: harnesses get thinner over time 建筑脚手架是临时设施，让工人搭建本来够不到的楼层，但它自己不是楼——楼建完就拆。随着模型变强，harness 的复杂度应该下降。Manus 在 6 个月里被重写 5 次，每次都在移除复杂度——复杂的工具定义变成了通用 shell 执行。 Scaffolding is temporary — it lets workers reach floors they couldn't otherwise, but it isn't the building, and it comes down when the building is done. As models get stronger, harness complexity should decrease. Manus was rewritten 5 times in 6 months, each time removing complexity — bespoke tool definitions became a generic shell exec.

2.2 Task 规范：4 组件

2.2 Task Spec: 4 Components

澄清：这 4 组件不是作者发明的，但其中一条约束才是论文真正的方法学贡献。 Clarification: the 4 components aren't invented here, but one constraint within them IS the paper's real methodological contribution. 这 4 组件（Instruction / Environment / Oracle / Verifier）沿用了 Harbor framework —— Terminal-Bench 的底层 containerized benchmark framework。SkillsBench 相对 Terminal-Bench 的方法学关键差异只多了一条小约束：Instruction 不能写明该用哪个 Skill（见 §2.4 "Skill Generality"）—— agent 必须自己从 skills/ 目录里发现、判断是否调用、怎么调用。这条约束把"Skill 效果"从 instruction-following 变成了可测量的独立变量，是 paired 评测（with-Skill vs no-Skill）成立的基础。

题外话，Oracle（预言机）在这里是算法设计的术语，不是"神谕"—— 指"存在即可证明任务可解"的参考解，用来保证"至少有一条路径能 pass verifier"。 The 4 components (Instruction / Environment / Oracle / Verifier) are inherited from the Harbor framework — the containerized-benchmark framework underlying Terminal-Bench. 2.4 领域与难度

2.4 Domains & Difficulty

澄清：task / trial / dataset 的区分（容易误读） Clarification: task vs. trial vs. dataset (common confusion)
  • 84 tasks 不是 dataset 大小——每个 task = 一个具体、单一的任务实例（例：mario-coin-counting = "这一个视频里数金币"；sec-financial-report = "这一次 Q2 vs Q3 2025 SEC 对比"）。不是 "给你 1000 张发票算平均"这种。
  • 84 tasks is NOT a dataset size — each task = one concrete, single-instance problem (e.g. mario-coin-counting = "count coins in this specific video"; sec-financial-report = "this specific Q2 vs. Q3 2025 SEC comparison"). Not "1000 invoices, report average accuracy".
  • 每个 task 下 agent 跑 5 次独立 trial，binary reward 取均值；总轨迹 84 × 5 × 3 条件 ≈ 每 config 1,092 runs（Table 7）。
  • Each task runs 5 independent trials, mean of binary rewards is task score; total ≈ 84 × 5 × 3 conditions ≈ 1,092 runs per config (Table 7).
  • 84 看起来少是刻意的：每个 task 是多步、domain-specific 的真实工作流（Extreme 级 26 个，人类 >4h），构建成本极高。322 候选 → 86 通过（26.7% 接受率）。论文 trade-off = 质量 / 现实性而非统计规模。
  • 84 looking small is intentional: each task is a multi-step, domain-specific real-world workflow (26 Extreme tasks need >4h from a human expert); 322 submissions yielded only 86 accepted (26.7% rate). Trade-off = quality / realism over statistical scale.
  • 规模对比：MMLU ~15k 题 / SWE-bench ~2.3k issue / BigCodeBench ~1.1k → SkillsBench 是"单元更重、数量更少"的类型。

Q4: Skill 与 RL options 的关系——Sutton 1999 的再现 Q4: Skills vs RL options — Sutton 1999 re-emerging

Option 是什么：来自 Sutton, Precup, Singh 1999 经典论文的概念——一段被"打包好的多步子策略"，agent 可以像调用函数一样调用它完成一段行为序列。例如机器人走迷宫，原子动作是 "上/下/左/右"，option 则是 "走到门口" / "打开门" / "穿过走廊"。每个 option 包含：(1) 什么时候可调用（initiation set）；(2) 内部子策略（policy）；(3) 什么时候终止（termination condition）。本质是 RL 里的"子程序 / function call"。

What's an option: a concept from Sutton, Precup, Singh 1999's classic paper — a "packaged multi-step sub-policy" that an agent can invoke like a function call to execute a sequence of actions. E.g. for a maze-navigating robot, primitive actions are "up/down/left/right"; an option might be "walk to door" / "open door" / "cross hallway". Each option has: (1) initiation set (when it can be called); (2) internal policy (what it does); (3) termination condition (when it stops). Essentially RL's "subroutine / function call".

Skill 和 option 的对应：两者都是 temporal abstraction——把多步行为打包成一个可调用的单元。本质抽象一样，实现不同：
