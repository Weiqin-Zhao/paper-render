Harness 到底是什么？

What exactly is a "harness"?

"Harness" 字面是马具/挽具，在 LLM 领域指的是围绕一个固定模型的所有代码——除了模型权重之外的一切。可以理解为：给你一个黑盒 model(prompt) → text，要把它变成"能解决具体任务的系统"时，你需要写的一切代码。

Literally "harness" = saddle/rigging. In the LLM world it means all the code wrapped around a fixed model — everything except the weights. Given a black-box model(prompt) → text, the harness is all the code you write to turn that into "a system that solves a concrete task."

Harness 里常见的成分
What a harness typically contains
  • Prompt 模板：system prompt、角色扮演、格式约束
  • Prompt template. System prompt, persona, format constraints
  • Retrieval 逻辑：从外部库查什么、怎么排序、塞几条
  • Retrieval logic. What to fetch from external stores, how to rank, how many to include
  • Memory / 状态：跨 turn 或跨 task 记什么、怎么更新
  • Memory / state. What to remember across turns/tasks, how to update
  • Control flow：tool-calling 循环、完成判定、错误重试
  • Control flow. Tool-calling loop, completion detection, retry logic
  • 后处理：解析、验证、re-ask 机制
  • Post-processing. Parse, validate, re-ask on failure
具象例子：Claude Code、Cursor、Cline
Concrete example: Claude Code, Cursor, Cline

这些 coding assistant 的底层模型都是 Claude/GPT，但 UX 和能力差异巨大——差的就是 harness：文件编辑 tool 的设计、context 压缩策略、补全展示方式、错误重试逻辑……同一个模型 + 不同 harness = 完全不同的产品体验。

Claude Code, Cursor, Cline all sit on top of Claude/GPT, yet their UX and capability differ enormously — the difference is the harness: file-edit tool design, context compaction, completion UI, retry logic... Same model + different harness = completely different product.

术语区分："Prompt engineering" 只关心单次 prompt 的写法；"harness engineering" 更宽，关心整个围绕模型的程序——包括动态 prompt 构造、检索、记忆、工具、控制流等。"Context engineering" 是近年兴起的中间词，强调"塞给模型的 context 怎么动态构造"这一子问题。本论文的研究对象是最广义的 harness。
Terminology. "Prompt engineering" concerns only how you write one prompt. "Harness engineering" is broader — the entire program around the model: dynamic prompt assembly, retrieval, memory, tools, control flow. "Context engineering" is a recent middle term focused on the subproblem of dynamically constructing what goes into context. This paper targets harness in the broadest sense.

Text optimizer 是什么？—— 一句话：把梯度下降搬到"文本/代码"上

What is a "text optimizer"? — One sentence: gradient descent, but over text/code

训练神经网络时我们有一个闭环：前向 → 算 loss → 反向得梯度 → 更新权重。但如果要优化的不是权重、而是一段 prompt 或 代码（它们都是离散文本，没有梯度），该怎么办？答案是让 LLM 自己扮演 optimizer：

When you train a neural net you have a loop: forward → compute loss → backward → update weights. But what if what you want to optimize is not weights but a prompt or a piece of code — discrete text with no gradient? The answer: let an LLM play the optimizer.

区分不同 text optimizer 的核心，其实就两个维度：

What distinguishes one text optimizer from another really comes down to two axes:

  • A. 优化对象 (artifact)：是 prompt、还是一小段代码、还是完整可执行 harness？
  • A. What's the artifact. A prompt? A small code snippet? A complete executable harness?
  • B. 反馈形式 (feedback channel)：传回去的是 scalar 分数？摘要？还是完整 trace？
  • B. What's the feedback channel. A scalar score? A summary? A full trace?
Meta-Harness 的论点就是沿着这两个维度做到极限：artifact 是完整可执行 harness（最大），反馈是整个文件系统（code + trace + score，最大）。几乎所有 prior 方法都在这两个维度上偷过某种懒。
Meta-Harness pushes both axes to the extreme: artifact = full executable harness (maximum); feedback = the entire filesystem (code + trace + score, maximum). Almost every prior method cuts corners on at least one of these axes.

1.1 Harness 为什么重要

1.1 Why harnesses matter

Tian 2026 SWEBench-MC 显示：在同一 benchmark 上只换 harness 不换模型，性能能差 6×。"harness"（围绕固定模型决定存/取/展示什么的代码）往往和模型本身同等重要。但 harness engineering 至今几乎全靠手工：人检查 failure、调启发式、迭代小规模设计。

Tian 2026 SWEBench-MC shows: changing only the harness around a fixed LLM can produce a 6× performance gap on the same benchmark. The harness — the code that decides what to store, retrieve, and show to the model — matters as much as the model itself. Yet harness engineering remains largely manual: inspect failures, tweak heuristics, iterate a handful of designs.

本文核心问题：这个过程本身能不能自动化？能不能构建一个外层系统，让它自己发现比人手工设计更好的 harness？
Core question. Can this process itself be automated? Can we build an outer-loop system that discovers better harnesses than hand design?

1.2 已有 text optimizer 为什么不够

1.2 Why existing text optimizers fall short

一眼看过去，harness 优化和文本优化很像（都是"根据过去尝试改 artifact"）。但现有方法有一个共同瓶颈：反馈被压缩得太狠——要么只看当前候选，要么主要传 scalar 分数，要么限定成短模板/摘要。而 harness 是 长程、有状态 的程序：某个"什么时候存什么"的决策可能几十步之后才影响 behavior。Summary 往往把追溯失败到早期 harness 决策所需的诊断线索扔掉。

Text optimization looks similar on the surface: iteratively improve an artifact from past attempts. But prior methods share one bottleneck — feedback is too compressed: they condition only on the current candidate, propagate mostly scalar scores, or restrict feedback to short templates / LLM summaries. Harnesses, however, are long-horizon, stateful programs: a single "what to store, when" decision can affect behavior many steps later. Summaries typically drop the diagnostic detail needed to trace downstream failures to earlier harness choices.

The paper quantifies this: across 6 representative text optimizers, available context per step is only 100–30k tokens — far below the diagnostic footprint required for harness search.

2.1 问题定义

2.1 Formulation

固定模型 $M$，任务分布 $\mathcal{X}$。Harness $H$ 是一个 stateful 程序，接受任务实例 $x \sim \mathcal{X}$，执行一条轨迹 $\tau \sim p_M(H, x)$，奖励函数 $r(\tau, x)$。目标：

Fix a model $M$ and task distribution $\mathcal{X}$. A harness $H$ is a stateful program; on task $x \sim \mathcal{X}$ it produces a trajectory $\tau \sim p_M(H, x)$ scored by $r(\tau, x)$. Objective:

$$H^* = \arg\max_{H} \; \mathbb{E}_{x \sim \mathcal{X},\; \tau \sim p_M(H, x)} \; r(\tau, x)$$

当同时关心多个指标时（例如 accuracy 和 context token 消耗），论文不用"0.7×acc − 0.3×cost"这种加权把它们捏成一个分数（即"不 scalarize"），而是用 Pareto 支配关系比大小：一个 harness "更好" 当且仅当它在所有维度上都不更差、且至少一个维度更好。最后只报告那条"既想精度更高、又不想 context 更多就没法再进一步"的 trade-off 曲线（Pareto 前沿）。

When multiple metrics matter (e.g. accuracy and context-token cost), the paper does not squash them into one number via weighting like "0.7×acc − 0.3×cost" (that would be "scalarizing"). Instead it uses Pareto dominance: one harness is "better" iff it's at least as good on every axis and strictly better on at least one. The reported result is the trade-off curve (Pareto frontier) — the points where you can't push accuracy up without also paying more context.

2.2 关键设计 1：Filesystem 作为反馈通道

2.2 Key design 1: Filesystem as feedback channel

不把历史写进 prompt，而是写进一个 proposer 能自主查询的文件系统。每个候选 harness 有自己的目录：

History is not packed into the prompt; it lives on a filesystem the proposer queries on demand. Each candidate gets its own directory:

  • Proposer 的"使用说明"（skill 文档）是另一份极小的固定文件，不在这个历史库里——它只告诉 proposer"新 harness 写到哪、用什么命令查历史、哪些文件不能碰"，整个 run 不变。
  • 智能来自于 proposer 跨多个 harness_00N/ 做横向对比和因果诊断，不是把经验浓缩成"learned rule" 给下一个候选用。
  • 这正是和 ACE / MCE 的关键区别：ACE / MCE 的"历史"是 curated 后的 rule / skill 库（proposer 只看浓缩过的经验）；Meta-Harness 的"历史"是原始尝试本身，由 proposer 自己读自己解读。
Clearing a common misconception. Each harness_00N/ contains the raw attempt record for one candidate harness (source + scores + trace). It is not a distilled skill document or guideline library.
  • The proposer's "instruction manual" (the skill doc) is a separate, tiny, fixed file that lives outside this history. It only says "write new harnesses here, query history with these commands, don't touch these files"; it doesn't change across iterations.
  • Intelligence comes from the proposer cross-reading many harness_00N/ directories and doing causal diagnosis, not from distilling experience into "learned rules" for the next candidate.
  • This is the key contrast with ACE / MCE: ACE / MCE's "history" is a curated rule / skill library (the proposer sees condensed experience); Meta-Harness's "history" is the raw attempts themselves, which the proposer reads and interprets first-hand.

Proposer（Claude Code）通过 grep、cat、ls 等标准终端工具检索，永远不把整个 filesystem 塞进 context。

The proposer (Claude Code) queries the store with standard terminal tools (grep, cat, ls); it never ingests the whole filesystem as a prompt.

实际用量：TerminalBench-2 run 中 proposer 每迭代中位读 82 个文件，跨 20+ 个先前候选。单次评估可产生 10M token 诊断信息。访问模式是 "非 Markov" 的——不是只看最近一个 parent（那会是 Markov 式的"只依赖前一步"），而是主动跨多个历史候选做横向对比。
Observed usage. On the TerminalBench-2 run the proposer reads a median of 82 files per iteration across 20+ prior candidates. A single evaluation can emit 10M tokens of diagnostic info. The access pattern is non-Markovian — not just conditioning on the latest parent (which would be Markov-style "only depends on the previous step"), but comparing across many historical candidates at once.

2.3 关键设计 2：Coding agent 而非 raw LLM

2.3 Key design 2: Coding agent rather than raw LLM

Proposer 必须是能执行工具调用 + 修改代码的 agent，三个原因：

The proposer must be an agent that can invoke tools and edit code, for three reasons:

  • 历史规模远超 context：必须主动决定 查什么。
  • History outgrows any context. The agent must decide what to inspect.
  • Harness 是可执行代码：需要写入/语法校验/运行等真实交互。
  • Harnesses are executable code. Writing, syntax-checking, and running code are first-class operations.
  • 对齐 coding agent 训练分布：read-write-execute 正是 frontier coding assistant 被重度训练的 workflow。
  • Aligned with coding-agent training. Read–write–execute is exactly what frontier coding assistants are trained on.
实现：Proposer = Claude Code + Opus-4.6（max reasoning）。只给一份极简 skill 文档说明目录布局和哪些文件可/不可改，不给诊断流程约束。
Implementation. Proposer = Claude Code + Opus-4.6 (max reasoning). Only a minimal skill document describes directory layout and read/write permissions — no diagnosis-flow prescription.

2.4 搜索循环（Algorithm 1）

2.4 Search loop (Algorithm 1)

先看人话版，再看下面的 pseudocode 就直观了：
  1. 准备阶段。扔几个"种子 harness"进去（zero-shot、few-shot、ACE 等已有 baseline）。每个都跑一遍、打分，把源码 + 分数 + trace一起写到一个文件夹里——这个文件夹就是"历史数据库" $\mathcal{D}$。
  2. 主循环（重复 N 次，典型值 N = 20）：
    1. 告诉 proposer（Claude Code + Opus-4.6）："你可以 grep / cat 这个文件夹里的一切，想看哪个候选的源码、想看哪条 trace 都行，然后给我写一个新的 harness。"
    2. Proposer 自己决定读什么（论文观察：每轮中位读 82 个文件），最后写出 k 个（典型 k=2–3）新 harness。
    3. 每个新 harness 先过一个超轻量的 interface check（能不能 import、签名对不对），过了才真正跑 eval（eval 是最贵的一步）。
    4. 跑完 eval 把（源码 + 分数 + trace）也丢进文件夹，下一轮 proposer 就能看到。
  3. 收尾。所有候选里挑出 Pareto 前沿（见 §2.1 解释：accuracy 或 context 没被任何别的候选同时击败的点集）作为最终结果。
关键省略：没有"选 parent、mutate、crossover"这些遗传算法套路——proposer 看什么、从哪个候选派生，全靠它自己。外层循环只负责"跑评估、落盘"。
Plain-English walkthrough first, then the pseudocode reads directly:
  1. Setup. Drop a handful of "seed harnesses" in (zero-shot, few-shot, ACE — existing baselines). Run each, score, write code + score + trace to a folder. That folder is our "history database" $\mathcal{D}$.
  2. Main loop (repeat N times, typically N = 20):
    1. Tell the proposer (Claude Code + Opus-4.6): "You can grep / cat anything in this folder — inspect any prior candidate's source, read any trace — and then write me a new harness."
    2. The proposer picks what to read on its own (paper measures: median 82 files/iter). It then emits k (typically k=2–3) new harnesses.
    3. Each new harness first passes an ultra-light interface check (does it import, does the signature match?); only if it passes do we run the full eval (eval is the expensive step).
    4. After eval, (code + score + trace) is dumped into the folder, so the next iteration's proposer sees it.
  3. Finish. Report the Pareto frontier across all candidates (see §2.1 — points not dominated on both accuracy and context).
What's deliberately missing: no parent selection, no mutation, no crossover — the proposer decides what to read and where to branch from entirely on its own. The outer loop only runs evals and writes logs.

常见疑问：为什么不直接保留分数最高的那个 harness，非要弄个 Pareto 前沿？ Common question: why not just keep the highest-scoring harness — why bother with a Pareto frontier?

这个答案是 two-part 的，取决于"你只有一个目标还是多个目标"。

The answer is two-part: it depends on whether you have one objective or several.

① 单目标场景——其实就是 argmax，论文确实也是这么做的

① Single-objective case — it really is just argmax, and that's exactly what the paper does

所以 math 场景就是按搜索集分数取 argmax——无需 Pareto。

③ 另一个相关但独立的设计：filesystem 里保留所有候选（含失败的）

③ A related but separate design: the filesystem keeps all candidates (including failures)

这不是"为什么用 Pareto"的理由（两件事在论文里是分开讲的），但顺便说一下：即便最终只报 1 个最优 harness，filesystem 里也必须保留所有历史候选。原因是 proposer 要靠这些失败 trace 做因果诊断。论文 §A.2 原话：

This is not a reason for using Pareto (the paper treats them as separate decisions), but worth noting: even if you only reported the single best harness, the filesystem still has to keep all prior candidates — because the proposer uses the failed traces for causal diagnosis. §A.2 quote:

"This kind of causal reasoning over prior failures is precisely what full-history filesystem access enables and what compressed-feedback optimizers cannot support."
一句话总结：Pareto 不是为了"不丢最优"，而是为了交出选择权给用户——让用户按自己实际的 accuracy/context 预算在曲线上挑点，而不是让作者替所有人决定权重。
One-liner: Pareto isn't about "not losing the best" — it's about handing the choice to the user, letting them pick a point on the curve according to their real accuracy/context budget, rather than the authors hardcoding a weighting for everyone.

设计思路：把单次 LLM 调用拆成两次小的，每次都只带很短的 context，总成本反而比一次性塞大量 few-shot 低。

Idea: split one LLM call into two small calls, each with a tiny prompt; total cost ends up lower than one big few-shot call.

  • Stage 1 (Draft)：从已积累的带标签 memory 里，按 TF-IDF 相似度取最像当前 query 的 5 个 例子 → 模型给出一个初步 label $D$。
  • Stage 1 (Draft). From memory of labeled examples, retrieve the 5 most TF-IDF-similar to the query → model produces a draft label $D$.
  • Stage 2 (Verify)：现在已经有 $D$ 了，第二次检索条件于 $D$ —— 拿 5 个"和 query 相似 且 标签 = $D$"的 confirmer + 5 个"和 query 相似 但 标签 ≠ $D$"的 challenger，让模型 keep 或 revise。
  • Stage 2 (Verify). Now that $D$ exists, second retrieval conditions on $D$ — fetch 5 confirmers (similar to query AND labeled = $D$) + 5 challengers (similar to query BUT labeled ≠ $D$), model keeps or revises.
  • 为什么这招关键：单次检索的 neighbors 只"最像 query"，但你真正想反驳 draft 的例子是"长得像 query 但偏偏不是 $D$ 的"——第二次检索才能拿到这种 targeted counterexample。
  • Why this matters: one-pass neighbors are only "most similar to query", but the examples that actually challenge the draft are "similar to query AND labeled ≠ $D$" — only the second retrieval can surface such targeted counterexamples.
  • 为什么便宜：两次调用各只带 ~5–10 个短样本，总 token 远小于一口气塞几十条 few-shot。
  • Why cheap: two calls each with only ~5–10 short examples; total tokens far below loading 30+ few-shot at once.
  • 冷启动：积累不到 5 个样本时 fallback 到普通 few-shot 单次调用。
  • Cold start: if <5 labeled examples accumulated, fall back to a plain single-call few-shot.
Label-Primed Query — 最准的那端（45.5k ctx, 48.6%）

设计思路：单次调用，但 prompt 刻意分三段，每段解决一个典型失败模式。

Idea: a single call, but the prompt is split into three blocks, each addressing a typical failure mode.

  • Label primer：在任何示例之前，先列出所有合法 label（LawBench 215 类、USPTO 180 类那种大类空间尤其重要）。为什么：模型不会编造不存在的 label，也保证它一开始就知道答案空间。
  • Label primer. Before any examples, list all valid labels (critical for LawBench's 215 classes or USPTO's 180). Why: prevents the model from hallucinating non-existent labels and grounds the answer space up front.
  • Coverage block：对每一个已见过的 label，从 memory 里挑 1 个"最贴 query"的样本放进来。为什么：即便是罕见 label，prompt 里也至少有一个例子，不至于被完全忽略。
  • Coverage block. For every label seen so far, pick 1 memory example most relevant to the query. Why: even rare labels get at least one example in prompt, so they're not forgotten.
  • Contrastive block：构造"长得很像 但 标签不同"的例子对，把这些对并排放进 prompt。为什么：这种对子直接向模型展示"哪些细微差别决定了 label 翻转"——局部决策边界。
  • Contrastive block. Build pairs of examples that are highly similar but labeled differently; place them side-by-side in the prompt. Why: these pairs directly expose "which subtle features flip the label" — the local decision boundary.
  • 检索规则：TF-IDF + query-anchored pairing —— 意思是 contrastive 对子不是全局找"最相似但不同 label 的两个"，而是以当前 query 为锚，找"query 附近"的那种对子。重点在这个 query 附近的决策边界，而不是全局最刁钻的对子。
  • Retrieval rule: TF-IDF + query-anchored pairing — contrastive pairs are not global "most similar but differently labeled" pairs; they're anchored to the current query, surfacing pairs from the query's local neighborhood. Focus is on the boundary near this query, not the globally trickiest pair.

5.2 数学检索 harness：4-route BM25 程序

5.2 Math retrieval harness: 4-route BM25 program

轻量词法路由器（关键词 + 正则）把每道题路由到 1 个 子策略；不 跨路由聚合（选定路由后其他路由不贡献任何例子）。所有阈值 / rerank 项 / per-route 数量都是 40 轮搜索自发涌现出来的，不是作者事后手调。

A lightweight lexical router (keywords + regex) routes each problem to exactly one sub-policy; no aggregation across routes (once a route is picked, the others contribute nothing). Every threshold, rerank term, and per-route size emerged from 40 rounds of search — not hand-tuned after the fact.

先解释每个技术词在干嘛，否则下面的流程图只是符号：
  • BM25@K：在 535k 题库上跑 BM25 关键词检索，保留得分最高的 K 条（即 top-K）。
  • Dedup to N：把 BM25 结果里"几乎重复的题"合并，最后留 N 条 不同题。
  • Rerank：按 BM25 打分再加一些额外因素重新排序。不同路由加的因素不同（见下）。
  • Keep K / Adaptive K：从 rerank 后的列表挑前 K 条放进 prompt。"Adaptive K"指 K 不是固定值，根据 top 几条分数集中程度动态决定（分数差距大→少取几条，分数差距小→多取几条）。
  • Hard NuminaMath reference：预先从 NuminaMath 筛出来的"难度 > 6"的题，在几何路由里当固定参考。
What each term means (otherwise the diagram below is just symbols):
  • BM25@K: run BM25 keyword retrieval on the 535k corpus, keep the top K scores.
  • Dedup to N: merge near-duplicates in the BM25 hits, keep N distinct problems.
  • Rerank: re-sort by BM25 score plus extra factors (which factors varies by route, see below).
  • Keep K / Adaptive K: take top K from the reranked list for the prompt. "Adaptive K" means K isn't fixed — it depends on how clustered the top scores are (tight scores → include more; big gaps → fewer).

各路由的"不一样"都有理由（这些是搜索涌现出来的 domain-specific insight，本身就是发现结果）：

Each route's quirks reflect search-discovered domain insight (these patterns themselves are part of the finding):

  • 组合（BM25@20 → dedup → rerank → keep 3）：宽召回 + 去重 + 挑难题重排——组合题解法多样性高，必须拉宽召回池，再 rerank 把"难例题"顶上来。
  • Combinatorics (BM25@20 → dedup → rerank → keep 3): wide recall + dedup + difficulty rerank. Solutions are highly diverse, so widen the pool, then rerank to surface harder examples.
  • 几何（1 hard ref + 2 raw BM25，不 rerank）：唯一不做 rerank 的路由。论文 §A.5 原话：搜索反复验证"几何题的原始词法匹配（比如 \triangle、\angle 这种符号对上）比按难度排序更有用"——符号结构对齐 > 难度偏好。
  • Geometry (1 hard ref + 2 raw BM25, no rerank): the only route that skips reranking. §A.5 quote: search repeatedly preferred raw lexical matches (e.g. matching \triangle, \angle tokens) over difficulty sorting — symbolic-structure alignment > difficulty preference here.
  • 数论（BM25@12 → rerank + 技术早示 bonus → keep 3）：独有"技术早示 bonus"——rerank 时给"解法前几行就明示 technique 的题"加分（比如开头就写 "Use LTE"、"Apply Fermat"）。道理：数论解法往往靠一个特定技巧打开，把这个技巧早早暴露的例子比藏在中段的更有用作模板。
  • Number theory (BM25@12 → rerank + early-technique bonus → keep 3): the unique "early-technique bonus" gives extra weight in rerank to solutions that state a technique up front (e.g. opening line "Use LTE", "Apply Fermat"). Why: NT solutions hinge on one key trick; examples that surface it early are more useful as templates than those burying it mid-proof.
  • 代数/其他（BM25@10 → rerank → adaptive K）：兜底路由，用 adaptive K 处理"多样题型"——分数集中就多取、分散就少取。
  • Algebra / other (BM25@10 → rerank → adaptive K): the fallback route; adaptive K handles heterogeneous problems — more retrievals when scores cluster, fewer when they spread.
最终程序是 两条成功搜索谱系的自主 merge：一条贡献 geometry route，另一条贡献 combinatorics route。Proposer 自己看出两条谱系互补、主动把它们合到一个程序里——没有外部 crossover 算子。
The final program is an autonomous merge of two successful search lineages — one contributed geometry, the other combinatorics. The proposer noticed the two lineages were complementary and merged them itself — no external crossover operator.

6.1 值得肯定的设计

6.1 What's well-designed

"把反馈做大"本身是非平凡贡献 Making feedback bigger is itself a nontrivial contribution

很多 text optimizer 把"feedback size 小"当 feature（"我们只需要 scalar score"）；Meta-Harness 反其道把反馈做到 10M token，然后用 ablation 证明 raw trace 不可压缩。这个方向在 coding agent 足够强之前甚至不可执行——工程和科学同时成立。

Many text optimizers treat "small feedback" as a feature ("we only need scalar scores"). Meta-Harness runs the opposite direction to 10M tokens and proves via ablation that raw traces are incompressible. This direction isn't even executable until coding agents are strong enough — engineering and science align.

对齐 bitter lesson：刻意不加结构 Aligned with the bitter lesson: deliberately minimal structure

（"Bitter lesson" 是 Rich Sutton 2019 年的论文，核心观点：算力增长时，通用搜索 / 学习方法总是赢过人手工结构。所以作者故意不加 mutation、parent-selection、persistent memory 等人工规则，把决策完全交给 coding agent。作者明确说 "as coding agents become more capable, the method improves automatically"。在 2026 年看，这种"把能力增长留给 agent"的系统设计越发正确。）

("Bitter lesson" is Rich Sutton's 2019 essay whose thesis is: as compute grows, general search/learning methods always beat hand-engineered structure. So the authors deliberately omit mutation, parent-selection, persistent memory, and other hand rules — all decisions go to the coding agent. They explicitly state "as coding agents become more capable, the method improves automatically." Viewed from 2026, leaving capability growth to the agent is increasingly right.)

Ablation 设计干净 Clean ablation design

scores-only / scores+summary / full-trace 把 "summary 是否足够" 这个自然反驳打得很实。summary 反而在 Best 上掉点（41.3 → 38.7），提示压缩本身是 negative information carrier。

scores-only / scores+summary / full-trace cleanly knocks down the natural counter-argument. Summary even hurts Best (41.3 → 38.7), hinting compression itself is a negative information carrier.

Qualitative trace 作为论据比纯 benchmark 更有说服力 Qualitative traces are more persuasive than benchmark numbers alone

§4 的 iter-by-iter 推理（"prompt 改动是 confound → 隔离变量 → pivot 到 additive 修改"）是 "filesystem 给了真诊断能力" 最硬的存在性证据。没有这段 log，ablation 数字只是相关性。

The iter-by-iter reasoning in §4 ("prompt edits are a confound → isolate variables → pivot to additive change") is the hardest existence proof for "filesystem enables real diagnosis". Without these logs, ablation numbers are only correlational.

6.2 可深入讨论的问题

6.2 Open questions

Cost 讨论完全缺失 Cost discussion is entirely missing

没有列出单次搜索 run 的成本（LLM API 费用 / wall clock / Claude Code 调用次数）。60 个 candidate × 每个 10M token proposer context + 每个评估的 base model cost，在 Opus-4.6 上粗估数千到上万美元一次 run。没有 USD/accuracy 曲线就无法判断"比手工贵多少"。

The paper never reports per-run cost (API USD, wall clock, Claude Code call count). 60 candidates × 10M-token proposer context + base-model eval cost on Opus-4.6 plausibly runs into the thousands-to-tens-of-thousands of USD per run. Without a USD/accuracy curve, "how much more expensive than hand-design" is unknowable.

Test set = Search set（TerminalBench-2）的严重性没被充分处理 Test-set = search-set (TerminalBench-2) is only lightly addressed

Discovery problem 的 framing 是公开实践没错，但"发现的 harness 过拟合到具体 89 任务的哪些 regex pattern"这个问题，靠人工 + 正则审计能真正挡住吗？作者自己承认 "resulting harness is specialized to TerminalBench-2 regime"——应该有一个独立的 held-out 测试。

The "discovery problem" framing is accepted community practice, but can manual + regex audits really catch all task-specific regex patterns that a harness might latch onto? The authors themselves admit "resulting harness is specialized to TerminalBench-2 regime" — an independent held-out test would be stronger evidence.

对 proposer 本身的依赖没有消融 No ablation over proposer capability

换成 GPT-5 或 Sonnet 4.6 还 work 吗？论文只说"依赖特定强 coding agent"，但没给出"proposer capability 到什么阈值这个 workflow 就破产"的曲线。

Would it work if we swapped in GPT-5 or Sonnet 4.6? The paper states "depends on a strong coding agent" but never plots "below what capability threshold does the workflow break".

反馈的 marginal value 递减曲线 Diminishing-returns curve for feedback size

从 scores → traces 有大跳（34.6 → 50.0），但从 5M → 10M tokens 呢？论文 claim "3 个数量级以上"，但没有 scaling 实验证明"越多越好"——只证了"比 100k 多得多"。

scores → traces is a large jump (34.6 → 50.0), but what about 5M → 10M tokens? The paper claims "3 orders of magnitude more" but shows no scaling experiment proving "more is strictly better" — only that it is "much more than 100k".

Harness = stateful program，但评估是跨 task 平均 Harnesses are stateful but evaluation averages over tasks

对于 online classification 这种真正 stateful 的 setting（memory 跨 task 累积），Pareto 结果是否对 task 顺序敏感？论文没讨论。

For truly stateful settings like online classification (memory accumulates across tasks), is the Pareto result sensitive to task ordering? Unaddressed.

安全与对齐问题 Safety and alignment concerns

一个 coding agent 自由读写 filesystem、跑任意 shell 命令（env bootstrap 本身就是例子）——在生产级 pipeline 里这是大 attack surface / 稳定性风险。文中只字未提。

A coding agent with free filesystem read/write and arbitrary shell (env bootstrap is itself an example) is a large attack surface in a production pipeline. Not discussed.

递归问题：Meta-Harness 能否优化自身？ Recursive question: can Meta-Harness optimize itself?

Meta-Harness 本身也是 harness（决定 proposer 看什么）。能不能用 Meta-Harness 优化 Meta-Harness？脚注里暗示这是更高阶的问题但没做实验。

Meta-Harness is itself a harness (it determines what the proposer sees). Could Meta-Harness optimize Meta-Harness? Hinted in a footnote but never experimentally tested.

6.4 Authors' practical tips (§A.8)

  • 写好 skill 文档：比调 iteration 数更重要。约束输出格式、安全行为，不约束诊断流程。准备 3–5 次 3–5 iteration 的调试 run 专门打磨 skill。
  • Write a good skill. Matters more than iteration count. Constrain output format and safety — not diagnosis flow. Budget 3–5 short (3–5-iter) debug runs just to polish the skill.
  • 起点要低、search set 要硬：对 baseline "易"的任务搜索无物可优化。50–100 样本足矣。
  • Start with a hard-for-baseline search set. If the baseline saturates the eval, there is nothing to search for. 50–100 examples is enough.
  • Log 机器可查询：JSON、层级、可读文件名、regex-friendly。
  • Make logs machine-queryable. JSON, hierarchical, readable filenames, regex-friendly.
  • 给 proposer 小 CLI（可选）：list_pareto / top_k / diff_runs。
  • Give the proposer a small CLI (optional): list_pareto / top_k / diff_runs.
  • 轻量 validation 先行：candidate 过 import + instantiate test 再跑完整 eval。
  • Lightweight validation first. Candidates pass import + instantiate test before full eval.
  • Eval 别让 proposer 做：独立 harness 跑 eval 写回 filesystem。
  • Don't let the proposer run evals. Separate harness scores and writes results back to the filesystem.
